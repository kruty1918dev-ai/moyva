# 0014 System Tech Debt Register

- Status: Accepted
- Date: 2026-05-11

## Context

Техборг накопичувався як локальні TODO і ситуативні правки без єдиного реєстру, власників і пріоритизації.
Це ускладнювало планування, створювало хаотичну еволюцію коду і збільшувало ризик випадкових регресій.

## Decision

Ввести системний техборг-реєстр як backlog-items з пріоритизацією.

1. Реєстр ведеться централізовано у docs/architecture/tech-debt-register.md.
2. Кожен debt-item має owner, status, target і score.
3. Пріоритет розраховується за формулою:
- Priority = Impact * Risk + Urgency - Effort.
4. Реєстр автоматично перевіряється у CI на валідність структури і порядок сортування.

## Consequences

Positive:
- Техборг стає керованим backlog, а не набором випадкових задач.
- Легше планувати ітерації та контролювати технічні ризики.
- Прозорі правила для triage і release-перегляду.

Trade-offs:
- Потрібна дисципліна підтримки реєстру.
- Додатковий процесний overhead для оновлення полів item.

## Rollback / Alternative

Rollback:
- Прибрати реєстр і CI-перевірку, повернувшись до ad-hoc debt handling.

Alternative:
- Вести debt у зовнішній issue-tracker системі, залишивши в репозиторії лише посилання і summary.
