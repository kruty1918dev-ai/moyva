## Вибір промислових ресурсів

### Мотивація

Раніше інженер мав вводити ID ресурсу вручну текстовим полем. Це приводило до помилок:
- опечатки в ID;
- посилання на неіснуючі ресурси;
- важко спотикутися про доступні ресурси без гугління в каталозі.

### Нова функціональність

При редагуванні промислової будівлі в вкладці **Entity Settings** → **Buildings** праведеться випадаюче меню **Ресурс** замість текстового поля.

```
┌─ Ресурс ────────────┐  ┌──┐
│ [Пшениця v]         │  │🌾│ ← іконка ресурсу з Icon поля
└─────────────────────┘  └──┘
```

### Як це працює

1. **Завантаження ресурсів**: На старті редактора сканується весь проект на наявність активів типу `EconomyResourceDefinition`.
2. **Кеш**: Усі знайдені ресурси кешуються в пам'яті редактора.
3. **Dropdown**: При редагуванні будівлі показується dropdown зі всіма ID ресурсів.
4. **Іконка**: Поруч з назвою ресурсу показується його Icon (якщо він є в DefinitionЕ) у вигляді 36×36px кваадрата.
5. **Автоматичне оновлення**: Якщо ви додадите новий ресурс до проекту або зміните ID, достатньо закрити/переклюючити в редакторі вкладку **Entity Settings** для оновлення dropdown.

### Приклад

**What was before (текстове поле):**
```
Industrial Resource ID: [wood___________] ← можлива опечатка
```

**What is now (dropdown):**
```
Ресурс: [Деревина (wood) v] 🪵 ← ясно вибираємо, бачимо іконку
```

### Правила

- **Доступні лише для промислових будівель** (`Category = Industrial`);
- **Автоматично очищуються** при зміні категорії будівлі на Non-Industrial;
- **Для стін/ратуш**: поле діпейбл (сірий), бо ці будівлі не потребують ресурсів;
- **Для цивільних/військових категорій**: поле заблоковане для редагування (вам не потрібен ресурс).

## Потребування тайлів

### Мотивація

Деякі промислові будівлі залежать від навколишнього терену:
- `Лісопилка` потребує лісу поруч;
- `Шахта` потребує гірської породи;
- `Ферма` добре росте на грунті/травіаені;
- `Риболовство` потребує води близько.

Раніше це налаштовувалось в коді або не налаштовувалось взагалі. Тепер дизайнер може налаштувати це прямо в редакторі без доторкання до коду.

### Нова функціональність

У вкладці **Entity Settings** → **Buildings** появляється новий розділ **Вимоги до тайлів**:

```
┌─ Вимоги до тайлів ───────────────────────────┐
│  ☑ Потребує тайлів                          │
│                                              │
│  + Додати вимогу до тайла                   │
│                                              │
│  [forest] | Радіус: 5  | Мін. кількість: 3 │ [✕]
│  [grass]  | Радіус: 4  | Мін. кількість: 2 │ [✕]
└──────────────────────────────────────────────┘
```

### Структура `TileRequirementDefinition`

Кожна вимога містить:
- **TileId**: ID тайла (dropdown вибір з `TileRegistrySO`);
- **Radius**: Радіус в клітинках, в якому має знаходиться тайл (мін. 1);
- **MinimumTileCount**: Мінімальна кількість клітинок цього тайла в радіусі (мін. 1).

### Приклад налаштування

**Лісопилка:**
```
Потребує тайлів: ☑

Вимога 1:
  - Тайл: [forest]
  - Радіус: 5 клітинок
  - Мін. кількість: 3 клітинки

(Тобто: хоча б 3 клітинки лісу мають бути в радіусі 5 від лісопилки)
```

**Шахта:**
```
Потребує тайлів: ☑

Вимога 1:
  - Тайл: [mountain]
  - Радіус: 4
  - Мін. кількість: 1

Вимога 2:
  - Тайл: [stone_outcrop]
  - Радіус: 3
  - Мін. кількість: 2
```

### Як це працює (runtime)

Система розміщення будівель перед дозволом розміщення перевіряє:
1. Чи має будівля `RequiresTiles = true`?
2. Якщо так, для кожного `TileRequirement[i]`:
   - Лічить клітинки типу `TileId` в радіусі `Radius` від позиції будівлі;
   - Якщо `count >= MinimumTileCount` — OK;
   - Інакше — забороняє розміщення.

### Інтеграція

- **Editor preview**: Валідатор економіки перевіряє що `TileId` існує в `TileRegistrySO`.
- **Runtime**: `ConstructionService` використовую ці вимоги при перевірці можливості розміщення.
- **Помилки розміщення**: Гравець бачить UI-повідомлення типу "Не вистачає лісу поруч для лісопилки".

## Практичні приклади

### Приклад 1: Просто промисловий ресурс без тайлів

**Силовея спілка** виробляє `metal` з ресурсу `iron_ore`, але не залежить від тайлів:

```yaml
Building: Forge
Category: Industrial
Industrial Resource: [iron_ore] 🔴
Requires Tiles: ☐ (unchecked)
```

### Приклад 2: Промисловий ресурс + потребування тайлів

**Лісопилка** виробляє `planks` з ресурсу `wood` і потребує лісу поруч:

```yaml
Building: Sawmill
Category: Industrial
Industrial Resource: [wood] 🌳
Requires Tiles: ☑
  - TileId: forest | Radius: 5 | MinCount: 3
  - TileId: wood_grove | Radius: 4 | MinCount: 1
```

### Приклад 3: Без промислового ресурсу, тільки тайли

**Мукомол** (не виробляє ресурс, а процесарює зб енергії від велика)

```yaml
Building: Watermill
Category: Industrial
Industrial Resource: (empty)
Requires Tiles: ☑
  - TileId: water | Radius: 3 | MinCount: 5
  - TileId: water_rapids | Radius: 2 | MinCount: 1
```

### Приклад 4: Гібридна конфігурація (Риболовецький дволи)

**Риболовецький дві** використовує ресурс `fish` і залежить від близькості води:

```yaml
Building: Fishery
Category: Industrial
Industrial Resource: [fish] 🐟
Requires Tiles: ☑
  - TileId: water | Radius: 4 | MinCount: 4
```

## Валідація

### Автоматичні перевірки

Вкладка **Validation** в редакторі автоматично перевіряє:

1. ✓ Кожен `TileId` в глубокій вимозі існує в `TileRegistrySO`.
2. ✓ Кожен `IndustrialResourceId` існує в `EconomyResourceDefinition` проекти (якщо non-empty).
3. ✓ `Radius >= 1` для всіх `TileRequirement[]`.
4. ✓ `MinimumTileCount >= 1` для всіх `TileRequirement[]`.
5. ✓ Вимоги до тайлів налаштовані лише для промислових будівель (попередження за інші категорії).

### Приклади помилок валідації

**Помилка 1: Невідомий тайл**
```
⚠ Building "Sawmill": TileRequirement[0] has unknown TileId "forest_dense"
  Fix: Choose "forest" or "forest_old_growth" from TileRegistry
```

**Помилка 2: Невідомий ресурс**
```
⚠ Building "Forge": IndustrialResourceId "iron_strange" does not exist in Resources
  Fix: Use one of: ["iron_ore", "copper_ore", "tin_ore"]
```

**Помилка 3: Невалідний радіус**
```
⚠ Building "Fishery": TileRequirement[0] has Radius=0, must be >= 1
  Fix: Change Radius to at least 1
```

### Auto-fix

Кнопка **Fix Common Issues** може автоматично виправити:
- Пустоту `TileId` на першу доступну з реєстру (з попередженням);
- Clamp значень: `Radius` = max(1, Radius), `MinCount` = max(1, MinCount).

## Як користуватися: Покроковий гайд

### 1. Додати промисловий ресурс

1. Відкрийте **Economy Designer** → вкладка **Entity Settings**.
2. В лівій панелі виберіть вашу промислову будівлю (або створіть нову).
3. В правій панелі розділ **Економіка**:
   - Поле **Ресурс**: натисніть dropdown, виберіть потрібний ресурс.
   - Іконка ресурсу появиться автоматично поруч.

### 2. Налаштувати потребування тайлів

1. Там же, в розділі **Вимоги до тайлів**:
   - Прапорець **Потребує тайлів**: включіть.
2. Натисніть **+ Додати вимогу до тайла**.
3. Для кожної вимоги:
   - **Тайл**: виберіть через dropdown з `TileRegistry`.
   - **Радіус**: встановіть в клітинки (рекомендовано 3-5).
   - **Мін. кількість**: скільки клітинок тайла потребується (рекомендовано 1-5).
4. Натисніть **Validate** для перевірки.

### 3. Перевірити через геймплей

1. Розмістіть будівлю на карті в редакторі сцени.
2. При спробі розміщення система перевірить вимоги.
3. Якщо тайли не недостатньо — розміщення буде заборонено.
4. Гравець/дизайнер побачить помилку розміщення з опиcom.

## FAQ

**Q: Мої новоі ресурси не з'являються в dropdown menuing?**
A: Перезагрузіть вкладку **Entity Settings** (переключіться на іншу вкладку і назад). Кеш оновлюється на `OnEnable`.

**Q: Можна лі встановити потребування до тайлів للеля non-industrial建筑?**
A: Технічно можна, але валідатор видасть warning. Рекомендується використовувати лише для Industrial категорії.

**Q: Який мінімальний радіус для потебування тайлів?**
A: Мінімум 1 клітинка. На практиці рекомендується 3-5 для реалістичності.

**Q: Що коли я видалю ресурс з проекту?**
A: Валідатор поінформує що `IndustrialResourceId` вказує на неіснуючий ресурс. Виправіть вручну або видаліть посилання.

**Q: Як мне налаштувати будівлю яка потребує кількох різних ресурсів?**
A: Наразі `IndustrialResourceId` одне. Для більш складних рецептів потібна розширена система рецептів (план на майбутнe). Поки що: якщо вам потрібна багатокомпонентна будівля, налаштуйте одного ресурсу як "головного", інші додайте через `EconomyProductionProfile` у вкладці `Production`.

## Інтеграція з іншими системами

### ConstructionService (검증розміщення)

При виклику `TryPreviewAt()` або `Confirm()` система:
1. Завантажує `BuildingDefinition` за ID будури.
2. Перевіляє `RequiresTiles`.
3. Для кожного `TileRequirement`: лічить тайли в радіусі.
4. Якщо умови задовольнені → позволяет розміщення.

### EconomyValidationService (Валідація редактора)

При запуску **Validation** перевіряються:
- Існування ресурсів і тайлів;
- Діапазони числів;
- Несумісные конфігурації (тайли для non-Industrial).

### TileRegistrySO (Словник тайлів)

Dropdown в UI лючеупдейтить з кошного `TileTypeDefinition.Id` з реєстру. При додаванні нового тайла він появиться в dropdown автоматично.

## Можливості розширень

1. **Підтримка кількох ресурсів**: Розширити `IndustrialResourceId` на масив `IndustrialResourceIds[]` з можливості вибору кількох ресурсів.
2. **Умовні вимогі**: Додати логік "ЯКщо тайл=forest АБО тайл=wood_grove" (будь я задоволениe).
3. **Вихідні вимоги**: Налаштувати де повинна знаходиться будівля (на тайлі, біля тайла, але не в центрі).
4. **Бонуси за тайли**: Якщо вокруг більше потрібних тайлів → виробництво шво більше.
