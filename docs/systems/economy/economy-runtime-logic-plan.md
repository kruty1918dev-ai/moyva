# Economy Runtime Logic Plan

## Мета
Зафіксувати узгоджені правила економіки та дати поетапний план реалізації runtime-системи для поселень, населення, виробництва, складів, караванів, торгівлі та споживання.

## Ключова вимога до інструменту
Усі правила економіки мають налаштовуватися в редакторі економіки.
`Economy Designer` переробляється в повноцінний `Economy Hub`:
- централізоване налаштування всіх параметрів (1-61);
- підтримка дефолтних значень + runtime override;
- валідація діапазонів і залежностей параметрів;
- preset-профілі для швидкого старту балансу.

## Вхідні домовленості з дизайну
Нижче зафіксовані правила, які вже узгоджені.

### Поселення
1. Поселення утворюється ратушею.
2. Ратуша є обов'язковою основою поселення.
3. Гравець не може знести ратушу; її може знищити лише ворог.
4. Після знищення ратуші будівлі лишаються, але стають некерованими.
5. Радіуси поселень не перетинаються; ратуші ставляться на мінімальній відстані n.
6. Росту Village -> Castle немає; замок окрема центральна точка.
7. Злиття поселень не відбувається (через обмеження радіусів).
8. Причини занепаду: дефіцит їжі/ресурсів, падіння населення, брак коштів.
9. Поселення без населення існувати не може.
10. Ліміт поселень: 3 (винести в параметр нової гри).

Початкові дефолти для хаба:
- MinTownHallDistance = 25 (налаштовується в редакторі)

### Населення
11. Ліміт населення задають житлові будинки.
12. Нові люди прибувають через n ходів; сім'я формується з 1 місцевого + 1 новоприбулого при наявності вільного житла.
13. Причини смерті: старість, голод, холод, хвороби, війна, руйнування будівлі (усі ваги та шанси конфігуруються).
14. Є професії та досвід; досвід підвищує продуктивність.
15. Будівля без робітників не працює (прийняти як базове правило).
16. Кількість робітників на будівлю задається в конфігу будівлі.
17. При дефіциті робітників застосовується пріоритет будівель (харчі > тепло > базові матеріали > інше).

Початкові дефолти для хаба:
- NewResidentsArrivalIntervalTurns = 10 (налаштовується в редакторі)

### Виробництво
18. Input/output задаються в конфігу будівлі; сировинні будівлі можуть працювати без input, але залежать від тайлів.
19. Без input виробництво повністю зупиняється.
20. Швидкість виробництва залежить від кількості робітників і їх досвіду.
21. Вплив тайлів є лише для тайл-залежних будівель.
22. Сезонності поки немає.
23. Псується лише їжа.
24. Склад безлімітний, зупинки через переповнення немає.
25. Одна будівля має один активний рецепт.

### Склади
26. Місткість безлімітна.
27. Амбар зберігає їжу, склад зберігає інші ресурси.
28. Переповнення відсутнє.
29. На поселення: 1 амбар + 1 склад.
30. Є спільний пул ресурсів поселення; вироблене одразу потрапляє в пул, input списується одразу з пулу.

### Каравани і логістика
31. Караванами керує гравець вручну.
32. Рух без дороги дозволений, але з меншим радіусом переміщення.
33. Дорога дає бонус радіусу переміщення всім юнітам.
34. Ворожий юніт може перехопити караван і отримати контроль над ним і вантажем.
35. Маршрути/доставка визначаються лише гравцем.
36. Якщо 2 каравани везуть один ресурс в одне місце, обидва додають ресурс у склад.
37. Пріоритет доставки визначає гравець.
38. Жорсткого ліміту дальності торгівлі немає.
39. Ліміт: 1 караван на 1 поселення.
40. При знищенні каравану лишається розбитий караван; ресурси можна забрати іншим караваном.

### Торгівля і ціни
41. Ціни ринкові.
42. Ціна залежить від обсягу ресурсу в поселенні та об'єму продажу.
43. Торгівля з нейтральними/ворожими фракціями поки вимкнена.
44. AI-торгівля поки не визначена (можна відкласти).
45. Більшість ресурсів прив'язані до біомів.

### Потреби і споживання
46. Споживання їжі залежить від віку; підтримуються всі вікові групи з окремими нормами.
47. Дефіцит їжі знижує hp і комфорт.
48. Додаткові потреби: дрова і одяг.
49. Усі жителі одного типу і споживають однаково.
50. Ресурсів розкоші немає.

### Будівлі
51. Maintenance відсутній.
52. Наслідків несплати maintenance немає.
53. Будівля може бути пошкоджена, але виробництво не зупиняє.
54. Апгрейдів будівель немає.
55. Виробнича залежність реалізується через ресурси (потрібен input), без жорсткої прямої зв'язки будівля->будівля.
56. Будівлю можна поставити на паузу.

### AI і автоматизація
57-61. На поточному етапі не реалізовувати автономну AI-економіку; залишити ручне керування гравцем і підготувати extensibility-точки.

### Уточнення за бойовою поведінкою караванів
- Перехоплення каравану має йти через бойову поведінку.
- До інтеграції повної бойової моделі використовується заглушка: миттєве перехоплення.

## Модель жителя (поглиблення, не лише цифри)
Жителі не є фізичними юнітами на мапі, але мають персональні картки в UI.

Поля жителя:
- Id
- Ім'я (генератор імен)
- Стать
- Вік
- Стан здоров'я (hp)
- Рівень комфорту
- Професія
- Досвід професії
- Шлюбний стан
- HouseholdId (будинок)

UI-вимога:
- При виборі будинку показати список мешканців цього будинку.

## Архітектурний план реалізації (Unity + Zenject)

### Етап 1. Domain-модель і конфіги
1. Додати runtime-моделі для:
- поселення
- домогосподарства
- жителя
- виробничого циклу
- складських пулів
- каравану і вантажу
2. Винести балансні параметри в ScriptableObject:
- споживання за віком
- формула народжуваності/приросту
- формула смерті від старості
- модифікатори комфорту
- формула ринкової ціни
3. Усі конфіги доступні через Economy Hub (без необхідності редагувати asset вручну в Inspector).
4. Параметри виносяться в окремі групи вкладок:
- Settlement Rules
- Population Rules
- Workforce Rules
- Production Rules
- Storage Rules
- Caravan Rules
- Market Rules
- Consumption/Needs Rules
- Building Rules
- AI Extensibility Rules
5. Підготувати enum і константи пріоритетів для розподілу робітників.

### Етап 2. Settlement Lifecycle Service
1. Сервіс створення/реєстрації поселення від ратуші.
2. Валідація мінімальної відстані між ратушами (non-overlap радіусів).
3. Обробка події знищення ратуші:
- статус поселення = Deactivated
- блок команд гравця
- блок податків/керування виробництвом
4. Конфігурований ліміт активних поселень (дефолт 3).

### Етап 3. Population Service
1. Tick-оновлення демографії кожні n ходів.
2. Прибуття нових жителів і спроба формування сімей.
3. Старіння, хвороби, голод/холод та інші причини смертності.
4. Призначення професій і накопичення професійного досвіду.
5. Розрахунок комфорту з факторів:
- їжа
- тепло
- світло
- податки
- закони

### Етап 4. Worker Allocation Service
1. Розподіл робітників по будівлях щотик.
2. Правило: без робітників будівля не працює.
3. Алгоритм пріоритетів:
- Харчовий ланцюг
- Тепло/одяг
- Базові матеріали
- Інше
4. Повернення невикористаних робітників у пул поселення.

### Етап 5. Production Service
1. Єдиний цикл перерахунку виробництва за тик.
2. Для кожної активної будівлі:
- перевірити паузу
- перевірити доступність робітників
- перевірити input-ресурси
- перевірити тайл-залежність
3. Якщо input відсутній -> повна зупинка цієї будівлі.
4. Вихідний об'єм множиться на модифікатор досвіду працівників.
5. Для їжі додати механіку псування за часом.

### Етап 6. Storage Service
1. Спільний ресурсний пул на поселення.
2. Логічне розділення на 2 контейнери:
- амбар (лише їжа)
- склад (решта)
3. Безлімітна місткість, але зі статистикою throughput для балансу.
4. Атомарні операції Reserve/Consume/Produce для уникнення гонок логіки.

### Етап 7. Caravan and Trade Service
1. Об'єкт каравану прив'язаний до settlementId.
2. Ліміт 1 караван на поселення.
3. Ручні накази гравця:
- завантажити
- перемістити
- продати/передати
- вивантажити
4. Модифікатор руху:
- без дороги: reduced range
- по дорозі: increased range
5. Бойові події:
- перехоплення ворогом
- знищення -> spawn wreck cargo entity
- підбір ресурсів іншим караваном
6. Ринкова ціна:
- базова ціна ресурсу
- коефіцієнт запасу в цільовому поселенні
- коефіцієнт об'єму угоди

Стартова формула ціни (вся параметризація редагується в Economy Hub):
- stockRatio = clamp(currentStock / targetStock, 0.1, 3.0)
- volumeRatio = clamp(tradeVolume / referenceVolume, 0.1, 3.0)
- scarcityFactor = pow(1.0 / stockRatio, stockExponent)
- volumeFactor = pow(volumeRatio, volumeExponent)
- finalPrice = basePrice * scarcityFactor * volumeFactor

Стартові дефолти коефіцієнтів:
- stockExponent = 0.35
- volumeExponent = 0.20
- minPriceMultiplier = 0.50
- maxPriceMultiplier = 2.50

Базові ціни ресурсів (дефолтний пресет, повністю налаштовується в Economy Hub):
- Wood = 5
- Stone = 6
- Grain = 5
- Meat = 8
- Berries = 6
- Water = 3
- Coal = 9
- Clothing = 12
- Planks = 8
- Tools = 14

### Етап 8. UI/UX економіки
1. Вікно поселення: ресурси, виробництво, демографія, комфорт.
2. Вікно будинку: список мешканців.
3. Картка жителя: ім'я, стать, вік, професія, досвід, hp, комфорт, шлюб.
4. Журнал подій: голод, холод, смерті, перехоплення каравану, дефіцити.

### Етап 9. Save/Load і тести
1. Зберігати повний стан економіки:
- поселення
- жителі
- професії/досвід
- склади
- каравани/вантажі/рештки караванів
2. EditMode тести:
- worker allocation
- production stop on missing input
- food decay
- market price formula
- settlement deactivation on townhall destroyed
3. PlayMode інтеграційні тести:
- цикл ходів із демографією
- ручна логістика караванів
- сценарій знищення/підбору каравану

## Невизначені питання, які треба зафіксувати перед кодингом
1. Значення n для:
- мінімальної відстані між ратушами (дефолт 25, налаштовується)
- періоду прибуття нових жителів (дефолт 10, налаштовується)
2. Розширення таблиці базових цін для всіх ресурсів проєкту (поза стартовим набором).
3. Калібрування дефолтних формул після перших плейтестів.
4. Майбутня інтеграція бойової поведінки каравану без заглушки.

## Дефолтні формули (стартовий пресет)
Усі коефіцієнти нижче змінюються в Economy Hub.

1. Комфорт жителя (0..100):
- comfort = foodScore * 0.35 + heatScore * 0.25 + lightScore * 0.10 + clothesScore * 0.10 + taxScore * 0.10 + lawScore * 0.10
- foodScore, heatScore, lightScore, clothesScore, taxScore, lawScore нормалізуються в діапазон 0..100

2. Смертність за хід (базова модель):
- baseDeathChance = ageComponent + hungerComponent + coldComponent + diseaseComponent + warComponent + collapseComponent
- ageComponent:
- 0.0000 (0-44 років)
- 0.0005 (45-54)
- 0.0015 (55-64)
- 0.0040 (65-74)
- 0.0100 (75+)
- hungerComponent = clamp(hungerSeverity * 0.015, 0, 0.15)
- coldComponent = clamp(coldSeverity * 0.010, 0, 0.10)
- diseaseComponent = clamp(diseaseSeverity * 0.012, 0, 0.12)
- warComponent = clamp(warSeverity * 0.020, 0, 0.20)
- collapseComponent = 1.0 при руйнуванні будинку (миттєва загибель)

3. Потреби і споживання за віком (за 1 хід):
- Child (0-15): Food = 0.6, Water = 0.7, Firewood = 0.4, Clothing = 0.2
- Adult (16-59): Food = 1.0, Water = 1.0, Firewood = 0.7, Clothing = 0.3
- Elder (60+): Food = 0.8, Water = 0.9, Firewood = 0.8, Clothing = 0.35

4. Штрафи дефіциту (старт):
- якщо Food або Water < 100% норми: hpDelta = -2 за хід, comfortDelta = -4 за хід
- якщо Firewood < 100% норми: hpDelta = -1 за хід, comfortDelta = -3 за хід
- якщо Clothing < 100% норми: comfortDelta = -2 за хід

## Матриця "все налаштовується в хабі"
1. Кожен пункт 1-61 має відповідний параметр, перемикач або формулу в Economy Hub.
2. Для кожної групи параметрів є:
- default preset
- пояснення в UI (tooltip)
- validation (min/max, required, logical dependency)
3. Жоден критичний економічний параметр не зашивається хардкодом у runtime-сервісах.
4. Runtime читає лише конфіг Economy Hub + збереження гри.

## Рекомендований порядок виконання (коротко)
1. Settlement + Population (ядро симуляції).
2. Worker Allocation + Production.
3. Storage + Food Decay.
4. Caravan + Market Pricing.
5. UI жителів і будинків.
6. Save/Load + тести.

## Критерії готовності першої версії
1. Можна створити до 3 поселень і симулювати ходи.
2. Населення зростає/старіє/помирає за правилами.
3. Виробництво залежить від робітників, input і досвіду.
4. Є ручна логістика караванів та ринкова торгівля.
5. При знищенні ратуші поселення деактивується.
6. При знищенні каравану ресурси не зникають безслідно.
7. Економічний стан коректно зберігається і завантажується.

## Developer Guide: Як додати новий тип будівлі

Нижче інструкція для розробників/контент-дизайнерів, як додати нову будівлю через Registry Hub так, щоб вона коректно працювала в економіці.

### Передумови
- Є доступ до `BuildingRegistrySO` у проєкті.
- Визначено, до якого класу належить нова будівля: `Civilian`, `Industrial`, `Military` або `Walls`.
- Якщо будівля `Industrial`, заздалегідь визначено `resourceId`.

### Крок 1. Відкрити Registry Hub
1. В Unity відкрийте вікно `Registry Hub`.
2. Переконайтеся, що завантажений правильний `BuildingRegistrySO`.

![Крок 1 - Відкриття Registry Hub](./images/add-building-type/step-1-open-registry-hub.png)

### Крок 2. Перейти у вкладку Buildings
1. У лівій панелі виберіть вкладку `Buildings`.
2. Знайдіть секцію `Створити нову будівлю`.

![Крок 2 - Вкладка Buildings](./images/add-building-type/step-2-open-buildings-tab.png)

### Крок 3. Заповнити базові поля
1. Заповніть `ID` (унікальний).
2. Заповніть `Display Name`.
3. Оберіть `Category`.
4. Призначте `Sprite`.
5. За потреби вкажіть `Prefab (override)` або залиште автогенерацію.

![Крок 3 - Базові поля будівлі](./images/add-building-type/step-3-fill-form.png)

### Крок 4. Заповнити економічні параметри
1. Вкажіть `Робітники` та `Пріоритет`.
2. Для `Civilian`:
- доступні `Житлова` і `Ліміт жителів`;
- доступний тип цивільної: `Regular`, `TownHall`, `Castle`.
3. Для `Industrial`:
- заповніть `Ресурс (resourceId)`;
- `Житлова` недоступна.
4. Для `Military` і `Walls`:
- житлові параметри вимкнені.

![Крок 4 - Економічні та класові параметри](./images/add-building-type/step-4-select-civilian-type.png)

### Крок 5. Створити запис і перевірити бейджі
1. Натисніть `Створити будівлю`.
2. У списку перевірте, що новий запис з'явився.
3. Для спеціальних типів перевірте бейджі:
- `TH` для ратуші;
- `CST` для замку;
- `R:<resourceId>` для індустріальних будівель.

![Крок 5 - Новий запис у списку](./images/add-building-type/step-5-created-row.png)

### Крок 6. Швидка runtime-перевірка
1. Запустіть сцену з економікою.
2. Розмістіть будівлю на мапі.
3. Перевірте очікувану поведінку:
- Civilian житло впливає на housing limit;
- Industrial працює через свій `resourceId`;
- Castle позначений як окремий центральний тип (`IsCastle`).

![Крок 6 - Перевірка у грі](./images/add-building-type/step-6-verify-in-game.png)

### Часті помилки
1. Дубльований `ID` -> запис не створюється.
2. Невірний або порожній `resourceId` для `Industrial` -> економічний ланцюг не зійдеться.
3. Непризначений `Prefab` -> у списку буде попередження.
4. Неправильна категорія -> приховані/недоступні очікувані поля.
