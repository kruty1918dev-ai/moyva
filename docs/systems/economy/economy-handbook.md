# Economy Handbook

## 1. Загальна структура документації

Рекомендована структура розділу economy:

- README.md: головний індекс, карта документів, швидка навігація.
- economy-handbook.md: цілісне пояснення системи (архітектура, runtime, multiplayer/bot, UI, setup).
- tutorial-add-resource.md: покроковий сценарій додавання нового ресурсу.
- economy-api-files-reference.md: пояснення ролі кожного файла в Economy/API.

Принцип: handbook дає загальне розуміння, tutorials дають кроки, API reference дає швидкі відповіді для розробки.

## 2. Загальне пояснення ігрової економіки

Економіка працює навколо поселень. Кожне поселення має:

- будівлі (житло, виробництво, інше);
- населення (жителі, вік, стан, потреби);
- спільний пул ресурсів (їжа, матеріали);
- правила (обмеження, формули, коефіцієнти).

Базовий цикл по ходу:

1. Оновлюється населення (прибуття, старіння, смертність).
2. Перерозподіляються робітники за пріоритетами.
3. Виконується виробництво (input -> output) з урахуванням робітників і профілів.
4. Оновлюються ресурси поселення і сигнали для UI/інших систем.

Ключовий принцип: критичні параметри не хардкодяться в runtime, а беруться з Economy Hub конфігів.

## 3. Навіщо потрібне кожне меню Economy Hub

Поточні вкладки (по EconomyDesignerWindow):

- Ресурси: каталог ресурсів, їх id, категорія (Food/Materials), базові атрибути.
- Оверайди: керування, які параметри можна змінювати перед грою.
- Правила: Поселення: дистанції ратуш, ліміти поселень, базові settlement-обмеження.
- Правила: Населення: темп приросту, демографічні коефіцієнти.
- Правила: Робоча сила: правила розподілу працівників.
- Правила: Виробництво: швидкість, цикли, залежності input/output.
- Правила: Склади: політики зберігання, резерви, логіка доступу.
- Правила: Каравани: ліміти караванів, вантажність, обмеження.
- Правила: Ринок: ціноутворення, коефіцієнти, у тому числі формати UI-підсумків.
- Правила: Споживання: норми витрат по групах.
- Правила: Смертність: модифікатори ризиків (голод, холод тощо).
- Правила: Будівлі: глобальні обмеження для будівельної логіки.
- Правила: ШІ: точки розширюваності AI-економіки.
- Валідація: перевірка цілісності конфігів + автофікс типових помилок.
- Симуляція: детерміністичний preview розрахунків для балансу.

## 4. Як це працює в мультиплеєрі та з ботом

Система owner-scoped: ресурси, події і агрегати розділені за ownerId.

Що це означає практично:

- Кожне поселення має OwnerId.
- Події економіки і будівництва несуть OwnerId.
- Агрегати для UI/логіки можна запитувати по ownerId.
- Контролер UI показує тільки дані поточного owner.

Для мультиплеєра:

- Два гравці в одній сесії не змішують економічні суми.
- Кожен клієнт/шар UI може запросити свої owner-тотали окремо.

Для бота:

- Бот отримує той самий API доступ по своєму ownerId.
- AI правила зараз підготовлені як extensibility, без повної автономної AI-економіки за замовчуванням.

## 5. Як це працює в runtime загалом

Основний runtime-потік:

1. EconomyInstaller робить DI-bind головних сервісів.
2. EconomyManager підписується на Calendar та Construction сигнали.
3. На BuildingPlacedSignal:
- якщо TownHall -> створюється поселення;
- інакше будівля додається в найближче поселення того ж owner.
4. На кожен ігровий хід EconomyManager запускає EconomyTickOrchestrator.
5. TickOrchestrator викликає підсервіси (population, workers, production, споживання/комфорт тощо).
6. Після тіку летять сигнали EconomyTickCompletedSignal, SettlementResourceChangedSignal тощо.
7. UI, save, AI та інші підписники реагують на сигнали.

## 6. Пояснення по UI

Поточний підсумковий UI-контролер:

- EconomyPlayerResourceSummaryUIController читає IEconomyRuntimeApi.
- Використовує TextMeshProUGUI поля для показу Materials/Food.
- owner береться автоматично з системи (IConstructionService.GetActiveOwner), а не з Inspector.
- Формат рядків береться централізовано з EconomyRulesConfiguration:
- ui-summary-materials-format
- ui-summary-food-format

Перевага: форматування і бізнес-логіка не дублюються в MonoBehaviour-компонентах.

## 7. Як сетапити (мінімально робоча конфігурація)

1. Створити EconomyDatabaseSO через Economy Hub.
2. Створити/прив'язати EconomyRulesConfigSO в базі.
3. Створити/прив'язати EconomyRulesConfiguration (шаблон параметрів) для Hub і runtime форматування.
4. Додати базові ResourceDefinition (мінімум 1 Food і 1 Materials).
5. Додати в сцену EconomyInstaller і призначити:
- database;
- rulesTemplate (рекомендовано).
6. Переконатися, що Calendar + SignalBus + ConstructionService присутні у сцені DI-контейнера.
7. Запустити Validation у Economy Hub.
8. Для UI: додати EconomyPlayerResourceSummaryUIController і прив'язати TMP-поля.

## 8. Що ще важливо (часто пропускають)

- Консистентність id: всі resourceId/buildingId мають бути уніфіковані без дублів.
- Migration/Schema: при зміні структури даних оновлювати міграції.
- Owner fallback: якщо owner не визначився, використовується player_0.
- UI-формати: шаблон рядка має містити {0}, інакше піде fallback формат.
- Validation перед PR: обов'язково проганяти перед комітом конфігів.
- Межі поточної реалізації: AI-економіка повністю автономна ще не увімкнена (extensibility stage).
