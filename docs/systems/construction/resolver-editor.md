# Construction: Resolver Editor і Універсальний Підбірник

← [Назад до Construction](../construction.md) · [Переглянути на сайті →](https://kruty1918dev-ai.github.io/moyva/#systems/construction/resolver-editor)

---

## Призначення

Цей модуль об'єднує дві частини в одну екосистему:

- **Resolver Editor**: редактор даних, де ви описуєте які типи існують і які варіації мають використовуватися.
- **Універсальний підбірник**: єдина runtime-точка входу, яка за контекстом мапи повертає підходящий `ID`.

Ключова ідея:

- дизайнер/розробник **описує** конфігурацію в редакторі;
- система **автоматично підбирає** потрібний `ID` у runtime;
- логіка не дублюється в різних сервісах.

---

## Що саме вирішує інструмент

- Прибирає розрізнені switch/if ланцюжки по проєкту.
- Дозволяє часткові реалізації: не обов'язково заповнювати всі типи.
- Підтримує варіативність: 1 варіант або декілька (рандомний вибір).
- Дає єдину декларативну модель `тип топології -> список ID`.

---

## Де знаходиться

### Runtime

- API підбірника: `IObjectTypePicker`
- Реалізація: `ObjectTypePickerService`
- Резолвер топології: `AutoTileVariantResolver`

### Editor

- Окреме вікно: `Moyva/Construction/Topology Resolver Editor`
- Вбудована панель у вікні: `Moyva/Construction/Wall Registry Editor`

---

## Підтримувані типи

У системі є централізований каталог типів (TopologyCaseType):

- Хрестове перехрестя
- Т-перехрестя: відкрито вгору/праворуч/вниз/ліворуч
- Кути: 4 варіанти
- Вертикальні: базовий, лівий акцент, правий акцент
- Горизонтальні: базовий, верхній акцент, нижній акцент
- Закінчення: 4 напрямки
- Діагоналі: 2 спеціальні випадки
- Ізольований

Важливо: якщо певний тип не налаштований, система переходить на fallback або повертає "нема збігу" без помилки.

---

## Семантика кутів і атласу

У проєкті назва кутового типу може означати позицію в атласі, а не прямий логічний конектор.

Приклад:

- арт-слот "правий верхній" може логічно відповідати з'єднанню зліва + знизу.

Це враховано на рівні мапінгу у runtime. У редакторі ви задаєте типи і варіації декларативно, а runtime застосовує правильну семантику.

## Швидкий старт (5 хв)

### Крок 1. Відкрийте редактор

Варіант A:

- `Moyva/Construction/Topology Resolver Editor`

Варіант B:

- `Moyva/Construction/Wall Registry Editor`
- розгорніть блок **Вбудований модуль резолвера**

### Крок 2. Оберіть реєстр

- Призначте `BuildingRegistrySO`.
- Якщо не призначено, натисніть автопошук у вікні.

### Крок 3. Оберіть колекцію

- Відкрийте потрібну колекцію стін.
- Якщо колекції немає, створіть її у Wall Registry.

### Крок 4. Додайте тип через "+"

- Натисніть кнопку **+** біля колекції.
- Оберіть тип із меню.
- Для кожного типу доступний детальний tooltip українською.

### Крок 5. Додайте варіацію

Для вибраного типу заповніть один із сценаріїв:

1. **Готовий prefab**:
   - задайте `Variant ID`
   - передайте prefab

2. **Лише sprite**:
   - задайте `Variant ID` (або лишіть порожнім для авто-ID)
   - передайте sprite
   - редактор створить prefab автоматично, збереже у вказаній папці і зареєструє в реєстрі

Після натискання "Додати":

- ID потрапляє у `TopologyBindings` для цього типу;
- за потреби створюється/оновлюється `BuildingDefinition`.

### Крок 6. Додайте додаткові варіанти (опціонально)

- Якщо для типу є 2+ ID, runtime обиратиме між ними випадково.
- Якщо один ID, використовується завжди.

### Крок 7. Перевірка в грі

- Запустіть Play Mode.
- Побудуйте стіни/сегменти.
- Переконайтесь, що підбір іде через нові дані з редактора.

## Детальний туторіал: створення повної стінової схеми

Ціль: налаштувати колекцію, де є лінії, кути, перехрестя і кінці.

1. У колекції додайте типи:
   - `Horizontal`, `Vertical`
   - `CornerNorthEast`, `CornerNorthWest`, `CornerSouthEast`, `CornerSouthWest`
   - `TJunctionOpenNorth`, `TJunctionOpenEast`, `TJunctionOpenSouth`, `TJunctionOpenWest`
   - `CrossIntersection`
   - `EndNorth`, `EndEast`, `EndSouth`, `EndWest`

2. Для бази додайте по 1 варіації на `Horizontal` і `Vertical`.

3. Для кутів додайте щонайменше по 1 варіації.

4. Для закінчень (`End*`) можна додати або власний арт, або тимчасово залишити fallback на лінійні сегменти.

5. Для `TJunction` і `CrossIntersection` додайте окремі варіації, якщо є арт; якщо ні, система використає fallback.

6. За потреби додайте по 2-3 варіації для найчастіших типів (`Horizontal`, `Vertical`) для візуальної різноманітності.

7. Перевірте runtime-сценарії:
   - пряма лінія
   - повороти
   - Т-перетини
   - хрест
   - кінцеві сегменти

## Як працює підбір у runtime

Пайплайн:

1. Система формує маску сусідів (`TopologyNeighborMask`) з контексту мапи.
2. `AutoTileVariantResolver` визначає найбільш релевантний `TopologyCaseType`.
3. `IObjectTypePicker` бере з колекції список ID для цього типу.
4. Якщо ID кілька, обирає один випадково.
5. Повертає фінальний `ID` викликаючій системі.

Fallback:

- якщо конкретний тип відсутній, пробується fallback-тип;
- якщо для fallback теж немає реалізації, повертається "нема збігу" без exception.

---

## Правила для командної роботи

- Усі нові типи додавайте через редактор, не через хардкод.
- Для кожного нового типу одразу додавайте хоча б один тестовий сценарій.
- Не дублюйте логіку підбору у Visual/Input/Service шарах.
- Якщо додаєте новий реєстр, робіть новий `IResolverRegistryAdapter`.

---

## Як розширити на інші реєстри

Редактор не прив'язаний до конкретного SO напряму. Для підключення нового реєстру:

1. Реалізуйте `IResolverRegistryAdapter`.
2. Додайте адаптер у `ResolverRegistryAdapterFactory`.
3. Переконайтесь, що адаптер повертає:
   - колекції,
   - ID колекції,
   - доступ до сериалізованих полів.

Після цього модуль редактора працюватиме з новим реєстром без копіювання UI-коду.

## Чеклист приймання

- Є сторінка і відкривається з docs index.
- Через "+" можна додавати типи у колекцію.
- Tooltip українською пояснює призначення типу.
- Можна додати варіацію через prefab.
- Можна додати варіацію через sprite з auto-prefab.
- Зміни відображаються в реєстрі одразу.
- Runtime повертає ID через єдиний підбірник.
- Часткові конфіги не ламають систему.

## Troubleshooting

### Не можу додати тип через "+"

- Перевірте, що в колекції ще є не додані типи.
- Якщо всі типи вже додані, меню покаже "Усі типи вже додані".

### Після додавання sprite не з'явився prefab

- Перевірте поле папки генерації.
- Перевірте права на запис в Assets.
- Перевірте Console на повідомлення AssetDatabase.

### Runtime не підбирає очікуваний ID

- Перевірте, чи тип присутній у `TopologyBindings` цієї колекції.
- Перевірте, що `sourceBuildingId` реально належить цій колекції.
- Перевірте семантику кутів (арт-позиція не завжди дорівнює логічному конектору).
