## Для кого цей документ

Цей документ написаний для дизайнерів, QA і технічних геймдизайнерів, які хочуть:
- подивитись, що збережено у слоті
- змінити конкретні дані без програмування
- прибрати пошкоджений блок
- відкотитись на backup
- видалити слот або весь файл

Документ пояснює всі терміни максимально просто.

## Де знаходиться інструмент

- `Moyva` → `Save System` → `Designer Tool`

Файл в коді:

## Що таке Save-файл (.mvs)

`.mvs` — це бінарний файл збереження.

Усередині є:
1. Заголовок (header)
2. Набір блоків (blocks)
3. Контрольна сума всього файлу (global CRC)

Спрощено:
- один файл може містити багато блоків
- кожен блок належить окремій підсистемі (наприклад, юніти, інвентар, fog)

---

## Словник термінів

### Byte (байт)
- Найменша одиниця даних у файлі.
- 1 байт = 8 біт.
- Значення байта: від 0 до 255.

### Binary (бінарний формат)
- Дані у вигляді байтів, а не "людського" тексту.
- Зручно для продуктивності та компактності.

### UTF-8
- Спосіб зберігання тексту у байтах.
- Якщо payload блоку насправді є текстом, його можна читати/редагувати як UTF-8.

### HEX (шістнадцятковий формат)
- Спосіб показувати байти для людини.
- Один байт у HEX виглядає як 2 символи, наприклад: `7F`, `00`, `A1`.
- HEX корисний, коли payload не є текстом.

### Payload
- Корисні дані блоку (вміст блоку).
- Саме це зазвичай хочуть редагувати.

### Block (блок)
- Окрема "секція" всередині `.mvs`.
- Блок має:
  - `blockId`
  - `blockSize`
  - `blockCrc`
  - `payload`

### BlockId
- Ідентифікатор блоку (число).
- Дозволяє системі зрозуміти, якому модулю належить блок.

### CRC / CRC32
- Контрольна сума для перевірки цілісності.
- Якщо CRC не сходиться, дані вважаються пошкодженими.

### Global CRC
- CRC для всього файлу (крім останніх 4 байтів footer).
- Якщо не сходиться, файл загалом вважається пошкодженим.

### Slot
- Номер save-слоту (`slot00.mvs`, `slot01.mvs`, ...).

### Backup (.bak)
- Резервна копія попереднього стану файлу.
- Використовується для відновлення після помилки запису.

### Temp (.tmp)
- Тимчасовий файл під час атомарного запису.
- Після успішного запису стає основним файлом.

## Файли в системі

Папка:
- `Application.persistentDataPath/saves/`

Типові файли:
- `slot00.mvs` — сейв слот 0
- `slot00.mvs.bak` — backup слот 0
- `slot00.mvs.tmp` — temp слот 0
- `config.mvs` — файл глобального конфігу

---

## Структура вікна Designer Tool

Інструмент ділиться на 4 логічні секції.

## 1) Вибір файлу

### Поле `Режим`
Варіанти:
- `Slot` — працюємо з конкретним слотом
- `Config` — працюємо з `config.mvs`
- `CustomPath` — вручну вибираємо будь-який `.mvs`

### Поле `Слот` (тільки в Slot)
- Діапазон: 0..99
- Формує файл `slotXX.mvs`

### Поле `Шлях` (тільки в CustomPath)
- Ручний шлях до файлу
- Кнопка `...` відкриває file picker

### Кнопка `Відкрити файл`
- Читає поточний вибраний файл
- Парсить заголовок, блоки, CRC

### Кнопка `Відкрити .bak`
- Відкриває backup для поточного файлу
- Зручно для перевірки/відновлення

### Кнопка `Відкрити директорію`
- Відкриває папку, в якій лежить поточний `.mvs` файл
- Зручно, якщо треба швидко знайти backup, temp або вручну зробити копію файла

### Кнопка `Видалити слот (файл + .bak + .tmp)`
- Працює тільки в режимі `Slot`
- Видаляє всі артефакти цього слоту

---

## 2) Інформація про файл

Поля:
- `Шлях` — де лежить файл
- `Версія` — версія формату `.mvs`
- `Розмір файлу` — фактичний розмір на диску
- `Кількість блоків` — скільки блоків у файлі
- `Global CRC` — OK або BAD

Можливий warning:
- файл обрізаний або має неповний блок

---

## 3) Список блоків

Для кожного блоку показано:
- `#index`
- `BlockId`
- `Payload size`
- `CRC status`

### Кнопка `Видалити` (біля блоку)
- Видаляє тільки цей блок

### Кнопка `Додати порожній блок`
- Додає блок без payload
- далі треба вручну задати `BlockId` і payload

---

## 4) Редактор блоку

### Поле `BlockId (uint)`
- Числовий ідентифікатор блоку
- Важливо: неправильний BlockId може зробити блок непотрібним для системи

### Редагування як UTF-8
- Зручно для простого текстового payload
- Кнопка `Застосувати UTF-8 → Payload` перетворює текст у байти
- Кнопка `Оновити UTF-8 з Payload` читає байти як текст

### Редагування як HEX
- Зручно для будь-яких бінарних payload
- Кнопка `Застосувати HEX → Payload` перетворює HEX у байти
- Кнопка `Оновити HEX з Payload` показує байти у HEX

Підтримуються розділювачі:
- пробіли, переноси, таби, `-`, `_`

## 5) Розумний перегляд блока генератора

Якщо вибраний блок належить `GeneratedWorldSaveModule`, вікно додатково показує:
- розмір карти (`width x height`)
- список унікальних біомів
- кількість статичних об'єктів карти
- список objectId з координатами
- приклад значення висоти з `heightMap`

Це потрібно, щоб дизайнер міг перевірити save генератора без читання сирого HEX.

---

## Глобальні кнопки (над секціями)

### `Зберегти поточний файл`
- Пересобирає `.mvs`
- Перераховує CRC блоків і global CRC
- Перезаписує файл

### `Видалити поточний файл`
- Видаляє відкритий файл повністю

### `Очистити всі блоки`
- Видаляє всі блоки в оперативній копії
- Потрібно натиснути `Зберегти`, щоб зміна записалась на диск

## Практичні сценарії для дизайнера

## Сценарій 1: Подивитись, що в слоті
1. Відкрити `Moyva/Save System/Designer Tool`
2. Режим: `Slot`
3. Вибрати номер
4. Натиснути `Відкрити файл`
5. Перевірити `Global CRC` і список блоків

## Сценарій 2: Прибрати пошкоджений блок
1. Відкрити файл
2. Знайти блок із CRC BAD
3. Натиснути `Видалити` біля цього блоку
4. Натиснути `Зберегти поточний файл`

## Сценарій 3: Відкотитись на backup
1. Відкрити `.bak`
2. Якщо backup коректний, зберегти його як основний файл через зовнішню копію
3. Або видалити основний файл і перейменувати `.bak` вручну

## Сценарій 4: Скинути один слот повністю
1. Режим `Slot`
2. Вибрати номер
3. Натиснути `Видалити слот (файл + .bak + .tmp)`

## Сценарій 5: Точкова правка payload
1. Вибрати блок
2. Змінити HEX або UTF-8
3. Натиснути відповідну кнопку `Застосувати ... → Payload`
4. Натиснути `Зберегти поточний файл`

## Сценарій 6: Знайти файл у системі
1. Відкрити файл у Designer Tool
2. Натиснути `Відкрити директорію`
3. У файловому менеджері побачити основний `.mvs`, `.bak`, `.tmp`

---

## Важливі попередження

1. Не змінюй `BlockId`, якщо не впевнені на 100%.
2. Якщо payload не є текстом, не редагуй його в UTF-8.
3. Після будь-яких змін обов'язково роби smoke-test сцени.
4. Перед ризиковими правками зроби копію файлу.
5. Якщо `Global CRC` BAD одразу після відкриття, файл уже був пошкоджений до редагування.

---

## Що робити, якщо дані виглядають дивно

Ознаки:
- дуже велика stamina (`E+30`)
- `EndOfStream` під час load
- частина юнітів не спавниться

Дії:
1. Відкрити `.bak` і порівняти
2. Перевірити цілісність `Global CRC`
3. Якщо блок явно пошкоджений, видалити його і перезберегти
4. Якщо проблема системна, передати файл програмісту разом з описом кроків

## Технічна примітка про сумісність форматів юнітів

Система юнітів підтримує:
- legacy формат (без stamina)
- новий формат (зі stamina)

Тому старі сейви повинні завантажуватись без аварійного падіння.

---

## Технічна примітка про блок генератора

Окремий block генератора містить:
- `width`
- `height`
- карту біомів (`biomeMap`)
- карту об'єктів (`objectMap`)
- карту висот (`heightMap`)

Це означає:
- позиція кожного тайлу і кожного objectId відновлюється з самого блока
- якщо такого блока в слоті немає, bootstrap трактує це як нову гру і світ генерується заново
