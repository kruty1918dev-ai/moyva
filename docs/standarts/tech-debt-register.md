# Tech Debt Register

Мета: вести техборг як керований backlog, а не як хаотичний набір випадкових правок.

## Принцип

Кожен запис техборгу є backlog-item з пріоритетом, власником, статусом і цільовим релізом.

## Де зберігається

- Реєстр: docs/architecture/tech-debt-register.md
- Валідація: tools/quality/validate-tech-debt-register.py
- CI check: .github/workflows/tech-debt-register.yml

## Поля backlog-item

1. ID: стабільний ідентифікатор (формат TD-XXX).
2. Title: короткий опис проблеми.
3. Area: модуль або підсистема.
4. Impact: вплив на продукт (1..5).
5. Risk: ризик регресій/аварій (1..5).
6. Effort: оцінка трудомісткості (1..5).
7. Urgency: терміновість (1..3).
8. Priority: обчислюваний пріоритет.
9. Status: Planned/InProgress/Blocked/Done.
10. Owner: відповідальний.
11. Target: цільовий реліз або ітерація.
12. Link: посилання на issue/PR/doc.

## Формула пріоритизації

Priority = Impact * Risk + Urgency - Effort

- Більший Priority означає вищий пріоритет.
- Реєстр підтримується відсортованим за спаданням Priority.

## Правила життєвого циклу

1. Новий техборг додається лише як backlog-item у реєстр.
2. Зміна статусу без Owner і Target заборонена.
3. Записи Done не видаляються, а лишаються історією рішень.
4. Якщо debt-item розбитий, батьківський item переходить у Done тільки після закриття всіх дочірніх.

## Операційний ритм

1. Щотижневий triage техборгу.
2. Перед релізом обов'язковий перегляд top-priority items.
3. Кожен архітектурний ADR має оцінювати, чи створює новий debt-item.
